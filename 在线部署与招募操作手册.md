# 数字关系核查实验：在线部署与招募操作手册

## 一、推荐架构

推荐使用：

1. **GitHub 私有仓库**：保存代码、记录版本、在需要时回滚；不要保存正式实验数据或密钥。
2. **Railway Web Service**：运行本项目的 Node.js 服务端和浏览器实验页面。
3. **Railway Volume**：挂载到 `/app/data`，持久保存 SQLite 数据库。
4. **Railway HTTPS 域名**：向被试发放带匿名编号的专属实验链接。

GitHub Pages 只适合静态 HTML、CSS 和 JavaScript，不能运行本项目的 Node 服务端，也不能可靠保存 SQLite 数据，因此不要单独使用 GitHub Pages 开展正式实验。

若被试主要位于中国大陆、样本量较大或伦理审批要求数据留在校内，正式实验优先考虑清华校内服务器或国内云主机；本项目已有 Dockerfile，可以使用相同容器部署。

## 二、正式部署前

1. 确认实验逻辑、指导语和知情同意书已经通过导师及伦理审批。
2. 在本地运行条件校验，必须看到两个 PASS：

   ```bash
   node scripts/check-experiment.mjs
   node scripts/check-counterbalancing.mjs
   ```

3. 删除 `data/` 中的预测试数据库，或者为正式实验使用新的空持久化磁盘。
4. 冻结 `EXPERIMENT_VERSION` 和 `MATERIAL_SEED`。正式收数期间不要修改 trial 生成逻辑。
5. 准备至少 48 个连续编号：`P001` 至 `P048`。48 人构成一轮完整顺序平衡。

## 三、上传 GitHub

1. 在 GitHub 新建 **Private repository**。
2. 只上传 `online_experiment/` 文件夹中的程序文件。
3. 确认 `.gitignore` 生效，以下内容不得上传：
   - `data/*.sqlite3`
   - `.env`
   - 真实被试登记表
   - 管理员令牌和编号哈希盐值
4. 将正式收数版本打一个固定标签，例如 `v1.5.1-formal`。

## 四、Railway 部署

1. 在 Railway 新建 Project，选择 **Deploy from GitHub repo**，连接上述私有仓库。
2. 若仓库根目录不是本文件夹，将 Root Directory 设置为 `/online_experiment`。
3. Railway 会自动读取 `Dockerfile` 与 `railway.json`。
4. 在 Variables 中设置：

   ```text
   HOST=0.0.0.0
   DATA_DIR=/app/data
   ADMIN_TOKEN=一段至少32位的随机字符串
   SUBJECT_CODE_SALT=另一段至少32位且不同的随机字符串
   PUBLIC_ORIGIN=https://你的正式域名
   ```

   `PORT` 使用 Railway 自动提供的值，不要手动固定。

5. 给服务添加 Volume，挂载路径必须是：

   ```text
   /app/data
   ```

6. 生成 Railway Public Domain。获得 HTTPS 地址后，将该完整来源地址写入 `PUBLIC_ORIGIN`，不要带末尾斜杠。
7. 重新部署，并访问：

   ```text
   https://你的域名/api/health
   ```

   返回 `"ok": true` 才表示服务和数据库正常。
8. 为 Volume 开启每日备份；每个收数日结束后再导出一份 CSV 到本地加密存储。

## 五、被试招募与编号

推荐由研究者预先生成并发放匿名编号，不让被试自行创建编号。

每位被试收到一个专属链接：

```text
https://你的域名/?sid=P001
https://你的域名/?sid=P002
...
```

登记表只在研究团队本地保存，建议包含：

| 字段 | 用途 |
|---|---|
| subject_id | 匿名编号，例如 P001 |
| contact_channel | 邮件或招募平台账号；不要上传实验服务器 |
| link_sent_at | 发放时间 |
| status | 未发放 / 已发放 / 已开始 / 已完成 / 退出 / 排除 |
| completion_checked_at | 完成状态核对时间 |
| compensation_status | 未发放 / 已发放 |
| replacement_id | 退出后的同顺序组替补编号 |
| notes | 设备异常、重复参加等情况 |

编号应连续发放，不要跳号。某个编号退出后，如需维持相同顺序组，用“原编号 + 48”作为替补。例如 `P017` 的替补是 `P065`。

## 六、数据监控与导出

健康检查：

```text
https://你的域名/api/health
```

CSV 导出：

```text
https://你的域名/api/admin/export.csv?token=你的ADMIN_TOKEN
```

管理员链接不得发给被试，也不要放进公开文档。建议使用单独浏览器窗口导出，并在导出后清除该条浏览记录。

每天检查：

1. 服务是否在线。
2. session 数和 trial 数是否持续增加。
3. 已完成 session 是否具有 300 个正式 trial。
4. 是否出现重复编号、异常退出或大量缺失 trial。
5. 当日 CSV 是否已下载并加密备份。

## 七、正式收数流程

1. 先进行 5–10 人预测试，重点检查完成时长、7×7 条件的可读性、点击准确性和主观评分负担。
2. 预测试数据与正式数据分开保存。
3. 正式发布前清空数据库、更新实验版本并再次运行校验。
4. 分批发放编号，例如每次 10–20 人，确认数据完整后再发下一批。
5. 正式收数期间锁定代码版本，不开启自动部署；修改必须先停止招募、保存数据并创建新实验版本。
6. 收数结束后导出 CSV 和 SQLite 原始数据库，生成只读归档，并保存代码版本号和材料种子。

## 八、其他平台

| 方案 | 便利性 | 数据控制 | 是否适合当前程序 |
|---|---:|---:|---|
| GitHub Pages | 高 | 低 | 不适合，缺少服务端与持久数据库 |
| Railway + GitHub | 高 | 中高 | 最适合快速预测试和中小规模正式实验 |
| Render + Persistent Disk | 高 | 中高 | 可以，但持久磁盘需要付费服务 |
| JATOS | 中 | 高 | 适合实验室已有 JATOS 时使用，但当前服务端需要改造成 JATOS 接口 |
| 校内服务器 / 国内云主机 | 中低 | 最高 | 适合正式长期收数，需要服务器维护和 HTTPS 配置 |
